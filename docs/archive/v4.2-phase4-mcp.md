## Context

Phase 4 opened the v4.2 sprint's second theme: the **Agent Ecosystem**. Phases 1-3 shipped
ecosystem-idiomatic governance presets (Node, Python, Go); Phase 4 makes those same docs
discoverable at runtime by any MCP client. Phase 5 (Agent Skills, TMS-430) extends the
ecosystem with first-party Claude Code skill artifacts.

TMS-429 was scoped as read-only first cut: no write tools, no resource subscriptions,
no HTTP/SSE transport. The pure/wiring split (`utils/resources.ts` vs `commands/mcp.ts`)
kept the SDK surface isolated from the "which files are exposed" question and made the
discovery layer unit-testable without the SDK.

## Notes

- Resource URI allowlist is canonical: `customFiles` entries in `.cortexrc` are silently
  excluded if not on the allowlist. This is the load-bearing protection against `.cortexrc`
  edits that try to expose `.env`, secrets, or unrelated source files.
- Pre-existing flaky timeouts in `hooks.test.ts` and `integration.test.ts` (carried over
  from Phase 3) did not worsen — generous 10s timeouts on the new subprocess tests
  absorbed startup variance.
- Stdout pollution is the protocol's failure mode: a static grep test on `src/commands/mcp.ts`
  for `chalk`, `ora`, `console.log`, `process.stdout.write` is part of the suite. If a
  future change adds any of these to the MCP command path, the test fails.
- The SDK 1.29.0 spike (pre-implementation hello-world) cost ~15 min and confirmed the
  protocol handshake before the rest of the discovery wiring was built. Recommended for
  any future MCP-adjacent work that pulls in a new SDK version.
